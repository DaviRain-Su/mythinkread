# Phase 1: PRD — MyThinkRead 云阅读器

## 1. 产品概述

MyThinkRead 是一款去中心化云阅读平台，**专注于 AI 原生内容** —— 所有书籍均由 AI 创作或人机协作创作，彻底规避版权风险。内容通过 **IPFS + Arweave 混合存储**（热读走 IPFS/Cloudflare 边缘，正式发布 + Wiki + 音频自动归档到 Arweave 永久保存），并融合社交属性构建 AI 文学社区。核心差异点：**AI 原生内容**、**数据开放可导出**、**去中心化永久存储**、**Kami 美学呈现**。

> **最终形态**：用户打开 MyThinkRead，看到的不只是"又一本 AI 书"，而是一本**设计优雅（Kami）、可读可听（ElevenLabs）、知识活的（LLM Wiki）、永远不会消失（Arweave）** 的文学作品。读完后还能把整本"第二大脑"（个人 Wiki + 笔记 + 音频片段）导出带走，真正拥有自己的阅读遗产。

## 2. 目标用户

| 用户类型 | 需求 | 痛点 |
|---------|------|------|
| AI 创作者 | 用 AI 生成小说/书籍、发布获取反馈、获得收益 | 缺乏 AI 原生内容发布平台；AI 工具与发布平台割裂 |
| 人机协作创作者 | AI 初稿 + 人工深度编辑，追求质量 | 现有工具要么纯 AI（质量差）要么纯人工（效率低） |
|  avid 读者 | 发现 AI 好书、记录阅读、社交分享 | 微信读书数据封闭、第三方统计困难 |
| 社交型读者 | 共读讨论、书单分享、获取认同 | 缺乏开放的读书社交平台 |
| 数据主权用户 | 拥有并导出自己的阅读数据 | 平台锁定、数据无法迁移 |
| 美学追求者 | 内容要有"体面"的呈现，拒绝模板化 AI 丑 | AI 生成内容排版差、分享卡片low、导出文档丑 |

## 10. V3 知识宇宙战略（Karpathy LLM Wiki）

### 10.1 愿景
从"AI 原生文学超级生态"升级为 **"AI 原生文学 + 永久活知识宇宙"**——每本书不仅是静态文本，更是**活的、可演化的知识图谱**。

灵感来源：Andrej Karpathy 的 LLM Wiki 理念——LLM 做知识工程师，自动把原始文档编译成结构化 Markdown Wiki，知识会累积而非每次从零开始。

### 10.2 核心理念
- **LLM 是知识工程师**：自动编译、维护、更新 Wiki，人类只做最终把关
- **知识会累积**：每次 Q&A、批注、洞见都被"归档"回 Wiki，下次查询建立在更丰富的知识基础上
- **用户完全拥有**：纯 Markdown 文件 + 图片，可一键导出到 Obsidian，不锁在任何平台
- **永久活知识**：Wiki 本身也上 Arweave，任何人用 CID/TX ID 都能公开访问

### 10.3 产品形态

**创作者端（本书圣经 Wiki）**：
- 书籍发布后，LLM 自动把全文"编译"成结构化 Wiki：
  - `concepts/` — 核心概念解释页
  - `entities/` — 人物/组织/地点实体卡（含关系图谱）
  - `themes/` — 主题脉络、象征意义
  - `timeline/` — 情节时间线
  - `foreshadowing/` — 伏笔与呼应分析
  - `worldbuilding/` — 世界观设定 Bible
- 创作过程中 LLM 自动维护一致性（角色卡自动更新、时间线自动校验）
- 一键生成章节时，AI 自动引用 Wiki 中的设定，避免崩坏

**读者端（个人第二大脑）**：
- 边读边听时，长按任何文本 → 呼出**本书活 Wiki**
- 不是一次性回答，而是进入结构化知识空间：
  - 图谱视图：人物关系网、概念关联图
  - 反向链接：这个概念在哪些章节出现
  - 主题导航：按主题浏览全书
- 划线/批注 → LLM 自动把洞见写入 Wiki 的 `analyses/` 目录
- 读完后一键导出整本 Wiki（Markdown + 音频片段）到 Obsidian

**社交端（社区共创 Wiki）**：
- 每本书有"公开 Wiki"模式，社区共同维护
- LLM 自动合并冲突、lint 检查矛盾
- 书摘卡片可带 Wiki 卡片（概念解释 + 关系图谱）
- 共读圈子基于 Wiki 进行深度讨论（"你们注意到第三章伏笔和结尾的呼应了吗？"）

**数据主权**：
- 整本 Wiki 可一键导出为 Obsidian 格式（Markdown + YAML frontmatter + 图片）
- 通过 API 授权第三方 AI 直接加载你的阅读知识库
- 即使离开平台，Wiki 仍永久可用（Arweave + 本地导出）

### 10.4 技术实现

| 组件 | 技术 | 作用 |
|------|------|------|
| Wiki 编译器 | Claude / Grok / Qwen via Cloudflare AI Gateway | 把 EPUB 编译成 Markdown Wiki |
| Wiki 维护 | LLM 增量更新（定时 Cron Trigger） | 自动发现新连接、lint 矛盾、更新索引 |
| 向量记忆 | Cloudflare Vectorize | 存储全书向量，AI 续写/朗读时引用角色设定，避免崩坏 |
| 图谱渲染 | D3.js / Cytoscape.js | 前端展示人物关系网、概念关联图 |
| Wiki 存储 | Arweave (永久) + IPFS (热访问) | 每本书 Wiki 额外 ~2-5MB |
| 导出工具 | Markdown + YAML + 图片打包 | 一键导出 Obsidian 格式 |
| 呈现层 | Kami Design System | Wiki 页面、导出文档、书摘卡片统一视觉体系 |

### 10.5 存储成本
- 一本 20 万字小说的 Wiki：~2-5MB Markdown + 图片
- Arweave 成本：~0.02-0.05 USD（一次性永久）
- 平台补贴：创作者前 3 本书免费生成 Wiki + 免费音频生成
- 补贴策略：降低创作者冷启动门槛，平台通过后续交易/订阅回收成本

### 10.6 竞品终极对比
| 产品 | 局限 | MyThinkRead 终极形态 |
|------|------|---------------------|
| 微信读书 | AI 问书一次性回答，数据封闭，无法导出 | 永久活 Wiki，知识累积，完全开放导出 |
| ElevenReader | 纯听书，无知识管理 | 听书 + 活 Wiki + 知识图谱 |

### 10.7 实现路径
- **V1 (MVP)**: 纯阅读
- **V1.5**: ElevenLabs 音频 + Kami 书摘卡片 + 简单章节 Wiki
- **V2**: 完整同步听书 + Kami 完整呈现层 + 本书活 Wiki（概念/实体/主题）
- **V3**: 社区共创 Wiki + 个人第二大脑导出 + 实时角色对话 + 动态 Kami 主题

## 11. 验收标准

### MVP 验收
- [x] AI 创作者可完成 AI 生成-编辑-发布-管理的完整流程 ✅
- [~] 人机协作模式有效：AI 初稿后创作者必须编辑才能发布 ⚠️ 有AI比例标注，无强制编辑检测
- [x] AI 比例标注准确显示在书籍详情页 ✅
- [x] 读者可按 AI 比例筛选书籍 ✅
- [x] 读者可完成发现-阅读-批注-评论的完整流程 ✅
- [~] 书籍内容可通过 IPFS CID 公开访问 ⚠️ 有IPFS上传功能，需验证公开访问
- [x] 用户可导出个人阅读数据 ✅
- [x] 榜单系统可正常展示排序，支持 AI 比例筛选 ✅
- [x] 评论区支持实时互动 ⚠️ 基础评论功能OK，AI质量讨论引导未实现
- [x] 平台所有内容均为 AI 原生或人机协作创作 ✅

### V2 验收
- [ ] 创作者可一键生成全书音频
- [ ] 读者可同步听书（文本高亮跟随语音）
- [ ] 书摘卡片支持语音分享
- [ ] 音频文件永久存储在 Arweave
- [ ] 听书数据可导出（含时长、偏好）
- [ ] 书摘卡片、Wiki 页面、导出文档使用 Kami 美学设计
- [ ] 创作者可一键生成 Kami 风格 One-Pager 导读和 Portfolio
- [ ] 章节长文档 PDF 导出使用 Kami 设计语言

### V3 验收
- [ ] 每本书自动编译成结构化 LLM Wiki（concepts/entities/themes/timeline）
- [ ] 读者阅读时可呼出本书活 Wiki（图谱视图 + 反向链接）
- [ ] 划线/批注自动归档到 Wiki analyses/ 目录
- [ ] Wiki 可一键导出为 Obsidian 格式（Markdown + YAML + 图片）
- [ ] 社区可共同维护公开 Wiki（LLM 自动合并冲突）
- [ ] Wiki 内容永久存储在 Arweave，支持 CID 公开访问
- [ ] 通过 API 授权第三方 AI 加载阅读知识库
- [ ] 动态 Kami 主题：根据书籍类型自动切换配色和字体（科幻=深蓝+等宽字体，言情=暖粉+手写体）
